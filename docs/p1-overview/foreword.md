# 前言

OpenClaw 是面向多平台消息面的本地优先控制平面与网关，强调自托管、模型无关与工程可观测性。它统一了多渠道接入与智能体运行时，让你的设备与数据在同一网关中被可靠地管理与路由。

## 为什么需要这本书

从原型到生产，智能体系统面临的真实问题包括：
- 多平台消息入口的碎片化与一致性治理
- 并发会话的状态化管理与隔离
- 工具与权限的精细化控制与审计
- 记忆与上下文的跨渠道复用与削峰
- 模型路由与回退的工程化落地

这些议题在通用 LLM 框架中往往缺少系统级答案，而 OpenClaw 提供的是一套面向生产的“控制平面与网关”思路。本书旨在把这些工程化经验与架构取舍，系统化、可落地地交付给你。

## OpenClaw 的定义与边界

OpenClaw 的核心是一套“个人助手平台”：本地优先的控制平面（Gateway 只是控制平面，产品本体是助手）+ 智能体运行时（Pi agent runtime）+ 会话模型（sessions）+ 工具与媒体管线（browser/canvas/nodes/cron 等）+ 多端节点与 UI（WebChat、macOS/iOS/Android）。所有客户端与设备通过 WebSocket 接入同一控制平面，采用事件型协议、令牌认证与设备配对保障连接与安全。围绕这一平台，系统分层清晰：
```
交互层 → 网关层（Gateway） → 智能体运行时 → 能力层（工具/记忆/媒体/浏览器） → 基础设施（配置/日志/安全）
```
- 本地优先与可自托管：在你的自有硬件与环境运行，数据可控
- 模型无关：支持多模型与路由回退，不绑定单一供应商
- 状态化会话：以工作区与运行时承载上下文、记忆与工具
- 可观测性与可审计：日志、事件与错误根因是“一等公民”
- 安全默认：默认拒绝、显式允许与最小权限的执行策略

上述特性均来自官方文档与实现，聚焦生产与维护的可行性，而非只做原型演示。

## 核心工程理念

- 渐进复杂，简单起步：从单会话到多会话、再到多租户与分布式，复杂度由真实需求驱动
- 分层解耦，清晰边界：层间以明确接口通信，便于独立演进与测试
- 可观测性优先：关键决策点有日志，状态变化有事件，错误具备可溯根因
- 安全默认，显式放行：默认拒绝、显式允许与最小权限贯穿执行点
- 面向失败的弹性：假设“任何环节可能出错”，为关键路径设计降级方案

这些原则贯穿网关、运行时与能力层，是 OpenClaw 的工程基石。

---

## 与 AgentOS 理念的铺垫

仓库未以“AgentOS/智能体操作系统”作为正式术语进行模块化定义，但从广义理念看，OpenClaw可与典型 AgentOS 分层形成对照（概念性映射，非官方边界）：
- 控制平面：Gateway 网关、路由与接入治理
- 运行时层：智能体运行时与会话隔离（工作区/上下文/记忆/工具）
- 能力层：工具、记忆、媒体管线与浏览器自动化
- 驱动/渠道：WhatsApp/Telegram/Discord/iMessage/WebChat 等渠道适配
- 观测与安全：日志、事件、令牌认证与最小权限

这种理念铺垫有助于把 OpenClaw 放入更广阔的智能体系统坐标中，但请注意：这是概念上的“视角对应”，不是官方的产品命名或模块契约。
